# 心脏毒性预测数据集增强 — 方案与需求文档

**版本**: v1.0
**日期**: 2026-04-13
**用途**: 供Claude Code进行代码实现的需求规格说明

---

## 1. 项目目标

通过扩展DIQTA训练集的化学空间邻域，结合多源证据检索，为每个分子构建包含**数值证据、临床状态证据、文献上下文证据**的证据包（Evidence Bundle），最终形成可支撑下游大模型生成"结构→靶点→效应"推理链的高质量数据集。

**最终交付物**：一个结构化JSON数据集，每条记录包含一个分子的完整证据包，可直接作为LLM推理链生成的输入。

**评估锚点**：在CredibleMeds测试集上具备跨数据集的泛化能力与逻辑解释力。

---

## 2. 核心设计原则

1. **标签来源二元化**：DIQTA原始分子标签视为真值；相似分子标签完全由多源证据决定，证据不足即丢弃，不强行赋标签。
2. **结构相似≠毒性相似**：Tanimoto相似性仅作为化学空间采样工具，不作为标签继承依据。
3. **可解释密度优先于数据量**：宁可丢弃证据不足的分子，也不引入噪声污染推理链。
4. **方法学一致性**：理化性质计算优先使用ChEMBL官方字段，缺失时使用RDKit本地计算兜底，保证整个数据集的计算方法可追溯。

---

## 3. 数据处理双路径架构

### 3.1 路径A：DIQTA原始分子（标签已知）

```
DIQTA分子 (SMILES + 已知标签)
    ↓
[步骤1] 需要的理化性质获取 (ChEMBL优先 → RDKit兜底)，具体需要哪些理化性质从./config/features.json读取
    ↓
[步骤2] 靶点活性数据检索 (ChEMBL: hERG, Nav1.5, Cav1.2) 只是先存起来，后续使用
    ↓
[步骤3] 临床状态证据检索 (ChEMBL max_phase + FDA/EMA + 撤市原因)
    ↓
[步骤4] 文献证据检索 (ChEMBL关联文献，通过chemblid或者activity关联 + PubMed)
    ↓
[步骤5] 证据包JSON组装 (标签沿用DIQTA真值)
    ↓
输出: Evidence Bundle (with ground-truth label)
```

**核心任务**：围绕已知真值补充支撑性证据，构建推理链的"事实支撑层"。

### 3.2 路径B：相似性扩展分子（标签未知）

```
DIQTA分子 (SMILES)
    ↓
[步骤1] ChEMBL相似性检索 (Tanimoto ≥ 0.8, Morgan FP)
    ↓
候选分子池
    ↓
[步骤2] 理化性质补全 (ChEMBL优先 → RDKit兜底) （把 M2 从“兜底模式”改成“融合模式”）
    ↓
[步骤3] 多源证据检索
    ├── ChEMBL: 靶点活性 + max_phase
    ├── ClinicalTrials.gov: 临床试验记录
    ├── 专利数据库: 专利毒性声明
    ├── CardioTox: 心脏毒性数据库
    ├── eTox: 毒理数据库
    ├── DILIrank: 药物毒性评级
    └── PubMed: 全文，这你搞不定问我，我给你个API
    ↓
[步骤4] 证据充分性判定
    ├── 充分且明确 → 赋予标签 → 进入数据集
    └── 不足或矛盾 → 丢弃
    ↓
[步骤5] 证据包JSON组装
    ↓
输出: Evidence Bundle (with evidence-derived label) 或 丢弃
```

**核心任务**：扩展化学空间覆盖度，证据驱动标签生成。

## 4. 功能模块规格

### 4.1 模块M1: 相似性检索器 (SimilarityRetriever)

**输入**：DIQTA分子的SMILES列表
**输出**：每个原始分子对应的相似分子ChEMBL ID列表

**技术要求**：
- 使用 `chembl_webresource_client` 的 similarity endpoint，注意使用的时候不要被封了
- Tanimoto阈值：≥ 0.8（基于ChEMBL默认Morgan FP，radius=2，nBits=2048）
- 去重：多个DIQTA分子可能检索到同一个相似分子，需记录所有关联关系
- 排除规则：如果相似分子本身已在DIQTA中，标记为"已有"，不进入路径B

**输出结构**:
```json
{
  "query_smiles": "...",
  "query_chembl_id": "CHEMBL...",
  "similar_molecules": [
    {
      "chembl_id": "CHEMBL...",
      "smiles": "...",
      "tanimoto": 0.87,
      "already_in_diqta": false
    }
  ]
}
```

### 4.2 模块M2: 理化性质计算器 (PhysChemCalculator)

**输入**：SMILES或ChEMBL ID
**输出**：固定标准描述符集


请把下面的描述符和features.json里的整理合并一下

**固定标准描述符集**（共13项）：

| 描述符 | 优先来源 | 兜底方法 |
|---|---|---|
| 分子量 (MW) | ChEMBL `full_mwt` | RDKit `Descriptors.MolWt` |
| logP | ChEMBL `alogp` | RDKit `Crippen.MolLogP` |
| logD (pH=7.4) | **需外部工具** | 见下方说明 |
| TPSA | ChEMBL `psa` | RDKit `Descriptors.TPSA` |
| 氢键供体数 (HBD) | ChEMBL `hbd` | RDKit `Lipinski.NumHDonors` |
| 氢键受体数 (HBA) | ChEMBL `hba` | RDKit `Lipinski.NumHAcceptors` |
| 可旋转键数 | ChEMBL `rtb` | RDKit `Lipinski.NumRotatableBonds` |
| 芳香环数 | - | RDKit `Lipinski.NumAromaticRings` |
| 重原子数 | ChEMBL `heavy_atoms` | RDKit `Descriptors.HeaviestAtomCount` |
| 碱性中心pKa | **需外部工具** | 见下方说明 |
| 酸性中心pKa | **需外部工具** | 见下方说明 |
| Lipinski规则违反数 | ChEMBL `num_ro5_violations` | RDKit手动计算 |
| QED类药性评分 | ChEMBL `qed_weighted` | RDKit `QED.qed` |

**pKa与logD的实现方案**：
- 首选：`pkasolver` 开源包（基于Graph Neural Network）
- 次选：调用 ChemAxon cxcalc（如有许可）
- logD计算逻辑：已知pKa和logP后，通过Henderson-Hasselbalch方程在pH=7.4下推算

**来源标记要求**：每个描述符值需附带 `source` 字段，取值为 `"chembl"` / `"rdkit"` / `"pkasolver"` / `"calculated"`，用于可追溯性。

### 4.3 模块M3: 多源证据检索器 (EvidenceRetriever)

分为三类证据的独立子模块：

#### M3-A: 数值证据 (BioactivityRetriever)

**目标靶点**（按优先级）：
1. hERG (ChEMBL target ID: CHEMBL240) — 核心靶点
2. Nav1.5 (CHEMBL1971) — 次要靶点
3. Cav1.2 (CHEMBL1940) — 次要靶点

**检索字段**：
- `standard_type`: IC50 / Ki / Kd
- `standard_value` + `standard_units`
- `pchembl_value`
- `assay_description`
- `assay_type` (B=Binding, F=Functional)
- `document_chembl_id` (关联文献)
- `confidence_score`

**输出结构**:
```json
{
  "target": "hERG",
  "target_chembl_id": "CHEMBL240",
  "measurements": [
    {
      "type": "IC50",
      "value": 1.2,
      "units": "uM",
      "pchembl": 5.92,
      "assay_type": "B",
      "assay_description": "...",
      "document_chembl_id": "CHEMBL...",
      "confidence_score": 9
    }
  ]
}
```

#### M3-B: 状态与临床证据 (ClinicalStatusRetriever)

**数据源**：
1. ChEMBL: `max_phase`, `withdrawn_flag`, `withdrawn_reason`, `withdrawn_year`, `withdrawn_country`, `black_box_warning`
2. ClinicalTrials.gov: 通过 `pref_name` 或 `synonyms` 检索 API v2，提取与QT、心律失常、心脏毒性相关的记录
3. FDA Orange Book / Drug Label: 提取boxed warning中的心脏相关警告
4. DILIrank: 药物毒性评级
5. CardioTox: 心脏毒性专项数据库
6. eTox: 毒理数据库

**输出结构**:
```json
{
  "max_phase": 4,
  "approved": true,
  "withdrawn": {
    "flag": true,
    "year": 2005,
    "country": "US",
    "reason": "QT prolongation leading to Torsades de Pointes"
  },
  "black_box_warning": true,
  "clinical_trials": [
    {
      "nct_id": "NCT...",
      "title": "...",
      "status": "Completed",
      "qt_related": true,
      "summary": "..."
    }
  ],
  "external_db_flags": {
    "cardiotox": {"present": true, "risk_level": "high", "evidence": "..."},
    "etox": {"present": false},
    "dilirank": {"present": true, "severity": "Most-DILI-Concern"}
  },
  "fda_label_warnings": ["..."]
}
```

#### M3-C: 文献与上下文证据 (LiteratureRetriever)

**策略**：
1. 通过ChEMBL activity记录中的 `document_chembl_id` 获取直接关联文献
2. 通过PubMed E-utilities API，用 `pref_name` + 心脏毒性关键词（"hERG", "QT prolongation", "Torsades de Pointes", "cardiotoxicity", "arrhythmia"）联合检索
3. 专利检索：通过Google Patents API或SureChEMBL，检索涉及该分子的心脏毒性相关专利

**噪声预过滤要求**（重要）：
- 标题或摘要中必须出现至少一个心脏毒性相关关键词
- 摘要中必须明确提及该分子名称或其同义词
- 排除综述类文章（通过MeSH标签 `Review` 过滤）——V1先标记，V2再决定是否硬过滤

**输出结构**:
```json
{
  "pubmed_articles": [
    {
      "pmid": "12345678",
      "title": "...",
      "abstract": "...",
      "mesh_terms": ["..."],
      "is_review": false,
      "relevance_keywords_hit": ["hERG", "QT prolongation"],
      "molecule_mentioned": true,
      "publication_year": 2020
    }
  ],
  "patents": [
    {
      "patent_id": "...",
      "title": "...",
      "abstract_excerpt": "...",
      "cardiotox_related": true
    }
  ]
}
```

### 4.4 模块M4: 证据包组装器 (EvidenceBundleAssembler)

**输入**：M2 + M3-A + M3-B + M3-C 的所有输出
**输出**：一个统一结构的JSON记录

**统一输出Schema**：

```json
{
  "molecule_id": "CHEMBL123456",
  "smiles": "...",
  "inchi_key": "...",
  "pref_name": "...",
  "synonyms": ["..."],

  "parent_diqta_molecules": ["CHEMBL..."],
  "tanimoto_to_parents": [0.87],

  "label": {
    "value": "torsadogenic | non-torsadogenic | null",
    "source": "diqta_ground_truth | evidence_derived | undetermined",
```

### 4.5 模块M5: 证据充分性判定器 (EvidenceSufficiencyJudge)

**适用范围**：仅路径B的相似分子

**V1策略**（先跑数据再定规则）：
- 不做硬过滤，保留所有检索到的候选分子
- 为每个分子计算一组"证据密度指标"，供后续分析：

```json
{
  "evidence_density": {
    "has_herg_measurement": true,
    "has_clinical_phase_info": true,
    "has_withdrawal_info": false,
    "external_db_hit_count": 2,
    "pubmed_article_count": 5,
    "pubmed_cardiotox_relevant_count": 3,
    "patent_count": 1,
    "total_score": 7
  }
}
```

**V2待定**：根据V1跑出的数据分布，制定阈值规则（例如 `total_score >= 5` 且 `has_herg_measurement == true` 才赋标签）。

**当前标签赋予逻辑（V1临时规则）**：
- 若有明确的 `withdrawn_reason` 提及QT/TdP → `torsadogenic`, confidence=high
- 若有hERG IC50 < 10μM 且有PubMed文献支持 → `torsadogenic`, confidence=medium
- 若有CardioTox明确记录 → 采纳其标签, confidence=high
- 否则 → `undetermined`（不进入最终数据集，但保留在raw数据中）

### 4.6 模块M6: 冲突检测器 (ConflictDetector) — V2

**当前版本只做检测与标记，不做裁决**。

检测场景：
- A类体外高毒（hERG IC50 < 1μM）但B类无临床警告 → 标记 `conflict_A_vs_B`
- 不同文献给出相反结论 → 标记 `conflict_within_C`

输出字段添加到证据包的metadata中，供后续人工审查或V2版本的裁决逻辑使用。

## 9. 注意

1. **优先级**：先完成 M1 → M2 → M3-A → M4 的最小闭环，用5个DIQTA分子跑通，再扩展其他模块。
2. **缓存优先**：所有API调用必须经过缓存层，避免重复请求。ChEMBL和PubMed对高频请求有限流。
3. **容错性**：任何单个分子的检索失败不应中断整个pipeline，需记录失败原因到 `discarded.jsonl`。
4. **可重入性**：pipeline必须支持中断后从checkpoint恢复。
5. **日志粒度**：每个分子的处理过程需有结构化日志，便于后续追溯。
6. **单元测试**：每个模块至少覆盖"正常分子 + 边界分子（无hERG数据） + 异常输入"三种场景。
7. **数据**：在./data下，缺的原始数据跟我要，比如DIQTA数据，比如爬取的时候各种账号
8. **参考代码**：已经有代码实现了具体的功能在./scripts/，你可以直接拿来参考
